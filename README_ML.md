## 2. EDA и выводы

Проведён разбор сырых транзакций (party_rk, category_nm, merchant_type_code, merchant_nm, real_transaction_dttm): объёмы, пропуски, кардинальности категорий и MCC, распределения по времени. Цель — оценка пригодности данных для коллаборативной фильтрации.

**Выводы:**
- Пропуски в категориях (~9%) обрабатываются явно (unknown), без потери пользователей.
- Высокая размерность (категории, MCC, мерчанты) потребовала редукции (PCA) и кластеризации для масштабируемого поиска похожих.
- Поведенческие паттерны (доли по категориям, времени, MCC) устойчиво кодируются в вектор и сравниваются через косинусное сходство.

**Решение:** пайплайн на основе только транзакций (без анкетных полей): фичи (доли по категориям, MCC, времени, токенам мерчантов) → StandardScaler → PCA → KMeans → рекомендации по cosine similarity в PCA-пространстве.

## 3. Пайплайн и API

### Offline → online

1. **Сырые данные:** CSV транзакций (party_rk, category_nm, merchant_type_code, merchant_nm, real_transaction_dttm).
2. **Валидация:** в `build_features.py` — `category_nm` → fillna/strip → `"unknown"`; `merchant_type_code` → coerce → fillna(-1); `merchant_nm` → fillna(""). В `prepare_features` и `seed_clusters` — согласованная обработка unknown и COALESCE по MCC.
3. **Фичи:** агрегация по пользователю: доли по категориям (cat_*), MCC (mcc_* + mcc_other), часу (hour_*), дню недели (dow_*), weekend_ratio, merchant_entropy, TF-IDF по токенам merchant_nm (tok_*). Итого 221 фича → `user_features.parquet`.
4. **Обучение:** StandardScaler → PCA (50 компонент) → MiniBatchKMeans (30 кластеров). Артефакты: scaler, pca, kmeans, `user_clusters.parquet` (party_rk, pca_*, cluster, top_cats).
5. **Инференс:** `MLRecommender` загружает `user_clusters.parquet` при первом запросе; маппинг user_id → party_rk → PCA-вектор; cosine similarity по всем; топ-K по score. Холодный старт — fallback (случайная выборка по кластерам).

### Эндпоинты

- **GET /api/recommendations** — список рекомендованных user_id, опционально с полем `match_reason`.
- **GET /api/rank** — тот же список с ранжированием: для каждого кандидата — `score` (cosine similarity в PCA-пространстве) и `match_reason`. Порядок — по убыванию score.
- **Health** — проверка доступности сервиса и модели.

### Ранжирование (rank)

Порядок кандидатов определяется только cosine similarity между PCA-векторами целевого пользователя и кандидатов. Чем выше сходство финансового поведения, тем выше score и позиция в выдаче. К каждому элементу прилагается текстовое объяснение (см. ниже).

### Объяснение мэтча

Двухуровневая схема:

- **Офлайн (fallback):** `explain_match` — приближённое восстановление вклада групп фич из PCA-координат (SHAP-like). По доминирующей группе (категории, время, MCC, токены мерчантов) формируется шаблонная фраза на русском. Работает без внешних сервисов.
- **Онлайн:** на сервере развёрнут **Qwen** исключительно для генерации формулировок объяснения. При `LLM_ENABLED=true` и доступном `LLM_URL` вызывается `explain_match_async`: по доминирующим меткам строится промпт, Qwen возвращает короткий текст. Ответы кэшируются в Redis (ключ `llm:explain:...`, TTL 7 дней). При недоступности LLM или ошибке используется офлайн-вариант.

Итог: по умолчанию — формулировка от Qwen; при отказе LLM — детерминированное объяснение по фичам.

## 4. Признаки

| Группа             | Кол-во | Описание |
|--------------------|--------|----------|
| `tx_count_log`     | 1      | log(1 + число транзакций) |
| `cat_*`            | 36     | Доли трат по категории (category_nm), data-driven |
| `mcc_*`            | 51     | Доли по MCC (merchant_type_code): топ-50 + mcc_other |
| `hour_0..23`       | 24     | Доли транзакций по часу |
| `dow_0..6`         | 7      | Доли по дню недели |
| `weekend_ratio`    | 1      | Доля транзакций в выходные |
| `merchant_entropy` | 1      | log(1 + число уникальных MCC) |
| `tok_*`            | 100    | TF-IDF по токенам merchant_nm (топ-100) |
| **Итого**          | **221** | |

Категории и MCC извлекаются из данных; пропуски: category_nm → `unknown`, MCC → отдельная обработка (-1 / mcc_other).

**IDF в пайплайне:** как и заложено в EDA, IDF используется — но только для токенов мерчантов (`tok_*`): формула `log((n_users + 1) / (doc_freq + 1))`, фича = tf × idf. По категориям (`cat_*`) и MCC (`mcc_*`) применяются доли без взвешивания по редкости; добавление IDF по категориям (больший вес редким категориям) — возможное улучшение.

## 5. MLflow, DVC, валидация данных

**MLflow.** В `train_pca_kmeans.py` логируются параметры (n_components, n_clusters, random_state), метрики (MAP@10, дисперсия PCA, inertia) и артефакты (scaler, pca, kmeans). Tracking URI — `http://mlflow:5000`, эксперимент `pca_kmeans_training`. Сервис mlflow поднят в docker-compose (backend sqlite, artifact root в volume). Обеспечивается воспроизводимость запусков и версионирование артефактов.

**DVC.** Бинарные артефакты обучения в `models/` версионируются через DVC (`scaler.pkl.dvc`, `pca.pkl.dvc`, `kmeans.pkl.dvc`); сами файлы получаются по `dvc pull`. `user_clusters.parquet` генерируется на сервере (train/seed_clusters) и в git не попадает.

**Валидация данных.** Первый шаг пайплайна — явная обработка пропусков: константы `CATEGORY_UNKNOWN`, `MCC_UNKNOWN` в `build_features.py`; fillna и нормализация для category_nm, merchant_type_code, merchant_nm; в `prepare_features` и `seed_clusters` — согласованное приведение к unknown и COALESCE. Это сохраняет консистентность с обучением и не отбрасывает пользователей из-за пропусков.

## 6. Оценка модели: evaluate_map

### Запуск

Из корня проекта (требуется `models/user_clusters.parquet`):

```bash
python -m app.model.evaluation.evaluate_map --k 100 --sample 500
```

Скрипт загружает PCA-векторы и метки кластеров, для выборки пользователей строит ранжирование по cosine similarity в топ-K и считает MAP@K (релевантность = тот же кластер), средний cosine в топ-K и coverage (доля запросов, у которых в топ-K есть хотя бы один из того же кластера).

### Результаты

| Метрика               | Значение   |
|-----------------------|------------|
| MAP@100               | **0.5753** |
| Среднее cosine@100    | **0.7111** |
| Coverage@100          | **99.40%** |
| Оценено пользователей | 500        |
| Всего пользователей   | 22 016     |
| Кластеров             | 50         |

### Интерпретация

MAP@100 ≈ 0.58 существенно выше случайного baseline (~1/n_clusters): модель ставит пользователей из того же кластера выше в списке. Средний cosine в топ-100 ≈ 0.71 отражает высокое сходство рекомендуемых с целевым. Coverage ≈ 99.4% — у практически всех запросов в топ-100 есть релевантный кандидат (тот же кластер).

### Ограничения и перспективы внедрения (в т.ч. в банке)

**Ограничения модели и пайплайна:**

- **Нет явного ground truth:** релевантность в оценке задаётся по кластеру (тот же кластер = релевантный), а не по реальным действиям пользователей (лайки, свайпы, мэтчи). Для продакшена нужна метрика по обратной связи и, при возможности, A/B-тесты.
- **Холодный старт:** пользователи без транзакций или с малым числом попадают в fallback (случайная выборка по кластерам). Для банка важно явно выделять таких пользователей и при необходимости дополнять анкетными/демографическими сигналами или онбордингом.
- **Актуальность данных:** модель обучена на срезе транзакций; при внедрении в банк нужен регламент переобучения (периодичность, триггеры по объёму новых данных) и консистентность фичей между обучением и инференсом.
- **Масштабирование:** поиск по cosine по всем пользователям в памяти упирается в рост объёма; при росте базы — переход на приближённый поиск (например, FAISS/ANN) по PCA-векторам.
- **Аудит и регуляторика:** объяснения (SHAP-like + LLM) дают интерпретируемый текст, но для банковского контекста может потребоваться фиксация версий модели, логирование решений и соответствие внутренним требованиям к объяснимости и персональным данным.

**Возможные улучшения:**

- Введение IDF по категориям (вес редких категорий) и, при наличии лейблов, переход к метрикам по обратной связи (CTR, конверсия в мэтч) и переобучению по ним.
- Расширение фичей (сезонность, суммы/частота по сегментам) и эксперименты с числом компонент PCA и кластеров (silhouette, стабильность).
- Индекс для быстрого поиска (FAISS/ANN), пайплайн переобучения по расписанию и мониторинг дрейфа фичей и метрик.
- Явная политика холодного старта (анкета, правила, отдельная модель) и, при необходимости, калибровка под бизнес-ограничения (диверсификация выдачи, fairness).
